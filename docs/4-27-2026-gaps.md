# 2026-04-27 — ZER Silent Gap Audit (Bare-Metal + Hosted)

Comprehensive audit of where neither compile-time checks NOR runtime traps catch
unsafe behavior. "Silent" means the program runs through the bug and produces
wrong/corrupt output without any detectable failure mode.

This document is the result of three audit rounds (~10,000 lines of compiler
source read across emitter.c, checker.c, zercheck.c, ast.h, ir_lower.c). Pure
silent gaps only — loud failures (linker errors, runtime traps) are not gaps,
they're contract violations the user can fix.

## Tier 1: ORIGINAL SILENT GAPS (already known, in CLAUDE.md)

### Gap 1 — Stack overflow on bare-metal without MMU page guard (HIGH)

Hosted: OS catches via stack guard pages → SIGSEGV. NOT a gap.
Bare-metal Cortex-M / RISC-V without MPU configured: stack pointer decrements
past stack bottom into adjacent memory → silent corruption of heap, globals,
or another task's stack.

**Fix path:** stack canary + `@max_recursion(N)` annotation → ~25 hrs.

### Gap 2 — Generation counter wraparound (LOW theoretical)

`emitter.c:4341` — `_zer_pool_alloc`: gen is `uint32_t`. After 4B alloc/free
cycles per slot, gen wraps to 1 (skips 0 reserved for null). A stale Handle
with the OLD gen could match the NEW gen → silent UAF undetected at runtime.

Realistic: ~12 years per slot at 10 free/sec, ~127 years for embedded sensor.
For very-long-running servers it matters.

**Fix path:** 64-bit gen counter or 24+40 split → ~5 hrs.

### Gap 3 — Atomic ordering bugs pre-Session-G (HIGH temporal)

`@atomic_*` intrinsics emit raw GCC `__atomic_*` builtins with `SEQ_CST`. But
ZER doesn't yet enforce that lock-free patterns use correct ordering. Manual
relaxed atomics + missing barriers → silent reorder bugs.

Closes with Session G (System #30 Atomic Ordering, ~80 hrs, planned).

### Gap 4 — `*opaque` ghost handle without wrap+`--track-cptrs` (MEDIUM) `[CLOSED 2026-04-27]`

User-controlled. Documented architecture pattern (use ZER wrapper around C
functions, enable `--track-cptrs`). Without both: heuristic-only tracking, some
patterns slip through.

**Fix path:** documentation + wrap pattern. Already in docs.

## Tier 2: NEWLY-FOUND SILENT GAPS (this audit)

### Gap 5 — Pool/Slab/Ring race ISR-vs-main (HIGH)

**Codebase evidence:** `emitter.c:4336-4348` `_zer_pool_alloc`:
```c
for (uint32_t i = 0; i < capacity; i++) {
    if (!used[i]) {       // <-- ISR can fire here
        used[i] = 1;       // race: ISR may have already claimed slot i
        ...
```

**Race scenario:**
1. Main calls `pool.alloc()`, finds slot 0 free
2. ISR fires, calls `pool.alloc()`, finds slot 0 free, sets `used[0]=1`,
   returns Handle
3. Main resumes, sets `used[0]=1`, returns Handle to SAME slot
4. Both contexts now hold a Handle to slot 0 → silent double-allocation

**Why it's a real gap:**
- CLAUDE.md states `Pool` is "ISR-safe, no malloc" but the runtime is non-atomic
- `check_isr_ban` only catches `slab.alloc` (heap), NOT `pool.alloc`
- Same problem applies to `pool.free` racing with `pool.alloc`
- Same problem applies to Ring (`*head`, `*tail`, `*count` non-atomic)
- Same problem applies to Slab (its metadata arrays are not synchronized
  cross-context)

**Fix path:** atomic CAS on `used[]` slots + compile-time check that Pool isn't
shared between ISR and main without atomics annotation → ~30 hrs. OR fix CLAUDE.md
to say "Pool ISR-safe ONLY when not shared between ISR and main" + emit warning.

### Gap 6 — Union tag-then-value torn writes (MEDIUM)

**Codebase evidence:** `emitter.c:1213-1242` union variant assignment:
```c
_zer_up%d->_tag = %u;          // tag updated FIRST
_zer_up%d->%s = ...            // value updated SECOND (no fence)
```

**Race scenario:**
- Main writes `msg.sensor = val` → `_tag = SENSOR; sensor = val;`
- ISR fires between the stores
- ISR reads union as SENSOR variant but value field still holds OLD data of
  PREVIOUS variant
- Silent type confusion — old data interpreted as new variant's type

**Why it's a real gap:**
- Variants can have different memory layouts and sizes
- ISR sees "SENSOR active" but reads stale memory
- No memory barrier between tag and value writes

**Fix path:** Reverse order (value first, fence, tag second) OR detect cross-
ISR union access at compile time → ~5 hrs.

### Gap 7 — Arena offset race (MEDIUM)

**Codebase evidence:** `emitter.c:4564` `_zer_arena_alloc`:
```c
size_t off = (a->offset + align - 1) & ~(align - 1);
if (off + size > a->capacity) return (void*)0;
a->offset = off + size;        // non-atomic
```

**Race:** main and ISR both call `arena.alloc()`. Both compute `off` from same
`a->offset`, both return SAME pointer → silent overlapping allocation.

**Fix path:** Document Arena as single-threaded only OR add atomic CAS → ~10 hrs.

### Gap 8 — Async state struct ISR re-entry (MEDIUM)

**Codebase evidence:** `emitter.c:8313` async poll uses Duff's device:
```c
self->_zer_state = %d; return 0;    // state set, then return
case %d:;                            // resumed via switch jump table
```

**Race scenario:** Async task suspended (state=N). ISR fires, ISR calls poll
function on SAME async task → reads `_zer_state` mid-update → jumps to wrong
case label or executes garbage.

**Why it's a real gap:**
- No language-level prevention of "calling poll() from ISR on currently-running
  task"
- Async state assumed cooperatively scheduled, not preempted

**Fix path:** Document async task rules (one poll site only) + compile-time
check that poll() not called from ISR → ~15 hrs.

### Gap 9 — `@probe` freestanding fallback returns success unconditionally (HIGH)

**Codebase evidence:** `emitter.c:4495-4503`:
```c
#else /* freestanding — no signal/setjmp, @probe unavailable */
static _zer_opt_u32 _zer_probe(uintptr_t addr) {
    /* freestanding: no fault handler, direct read (same as C) */
    volatile uint32_t *p = (volatile uint32_t *)(uintptr_t)addr;
    uint32_t val = *p;
    return (_zer_opt_u32){ val, 1 };  // <-- always has_value=1, even on bad read
}
#endif
```

**Why it's a HIGH severity gap:**
- ZER spec says `@probe(addr)` returns `?u32` with `null` if address faults
- On freestanding (bare-metal without setjmp/signal): direct `*p` then return
  `{val, 1}`
- If address actually faults, hardware exception fires → embedded chip resets
  or jumps to fault handler
- If address points to garbage memory but doesn't fault: returns garbage `val`
  with `has_value=true` → caller thinks read succeeded
- **Semantic divergence between hosted and freestanding** — code that uses
  `@probe` for safe MMIO discovery silently fails on bare-metal

**Fix path:** Either disable `@probe` on freestanding (compile error) OR
document the divergence loudly. ~3 hrs to add compile error.

### Gap 10 — `@critical` on x86_64/unknown arch only emits memory fence (HIGH)

**Codebase evidence:** `emitter.c:8420-8444`:
```c
#if defined(__ARM_ARCH)
    cpsid i                     // disable interrupts
#elif defined(__AVR__)
    cli
#elif defined(__riscv)
    csrrci %0, mstatus, 8
#else
    __atomic_thread_fence(__ATOMIC_SEQ_CST);   // <-- NOT disabling interrupts!
#endif
```

**Why it's a HIGH severity gap:**
- On x86_64 hosted: user mode CLI/STI = SIGSEGV, so fence is the only safe
  fallback (correct behavior, interrupts can't be disabled in user mode anyway)
- On x86_64 BARE-METAL kernel/bootloader: `@critical` should emit `cli/sti`
  but instead emits a fence. Silent semantic divergence — user thinks they
  disabled interrupts, but didn't
- On unknown architectures: same fence-only fallback. User has no idea
  interrupts are still firing

**Fix path:** Detect bare-metal x86 build (`__STDC_HOSTED__ == 0` AND
`__x86_64__`) and emit `cli/sti`. Plus emit warning on unknown archs that
fence is not interrupt-disabling. ~5 hrs.

### Gap 11 — FPU context not auto-saved in interrupt handler (MEDIUM)

**Codebase evidence:** `emitter.c:4156`:
```c
emit(e, "void __attribute__((interrupt)) %.*s_IRQHandler(void);\n", ...);
```

**Bare-metal issue (Cortex-M4F, RISC-V with F):**
- GCC's `__attribute__((interrupt))` saves caller-saved INTEGER registers
- On Cortex-M4F: FPU registers may NOT be saved by default — depends on
  CONTROL.FPCA bit and lazy stacking config
- If main is mid-FPU computation and ISR uses any float operation → main's
  FPU state silently corrupted → wrong floating-point results

**Why it's a real gap:**
- ZER allows `f32`/`f64` types in any function including ISRs
- No compile-time check that ISR using FPU also saves/restores FPU state
- Cortex-M4F's lazy stacking is a runtime CPU feature, not language-level

**Fix path:** Detect FPU use in ISR body → emit `@cpu_save_fpu/@cpu_restore_fpu`
prologue/epilogue automatically OR warn user → ~10 hrs.

### Gap 12 — Packed struct as MMIO target → misaligned access (MEDIUM)

**Codebase evidence:** Packed struct alignment is caught for `@atomic_*`
(checker.c:6477) but NOT for direct field reads through
`@inttoptr(*PackedReg, 0xADDR)`.

**Bare-metal issue:**
- User declares `packed struct UART { u8 ctrl; u32 data; }`
- Points to MMIO with `@inttoptr(*UART, 0x4001_3800)`
- Reading `data` requires misaligned 32-bit load
- ARMv7-M (Cortex-M3+) supports unaligned access on normal memory but NOT on
  Device-mapped MMIO regions — bus error
- ARM Cortex-M0+: unaligned access = silent corruption (returns wrong value)

**Fix path:** Reject `packed struct` used as MMIO target type when fields are
non-aligned (when target type comes from `@inttoptr`) → ~5 hrs.

### Gap 13 — Auto-zero relies on .bss zeroing by linker/startup (LOW) `[CLOSED 2026-04-27 — docs]`

ZER auto-zeros all variables. Uninitialized globals end up in `.bss` section.

**Bare-metal issue:** Some bare-metal runtimes (especially custom linker
scripts) don't zero `.bss` automatically — user must implement startup code.
If `.bss` not zeroed, ZER's "everything auto-zeroed" guarantee silently broken
— variables hold whatever random values were in RAM at boot.

**Fix path:** Document the requirement OR emit a runtime check on entry that
walks `.bss` and zeros it → ~2 hrs.

### Gap 14 — `-fwrapv` not enforced when user invokes own gcc (LOW) `[CLOSED 2026-04-27]`

CLAUDE.md says "ZER defines signed overflow as wrapping. `zerc --run` adds
`-fwrapv` automatically."

**Issue:** When user invokes their own bare-metal cross-gcc on the emitted .c
file (typical embedded workflow with custom linker scripts), they may forget
`-fwrapv`. Then signed overflow becomes UB → optimizer can do anything → silent
miscompile.

**Fix path:** Emit `#pragma GCC optimize("wrapv")` in preamble OR clearly
document the requirement → ~2 hrs.

### Gap 15 — Function pointer call cannot trigger zercheck free tracking (HIGH)

**Codebase evidence:** `zercheck.c:242` `is_free_call`:
```c
if (!callee || callee->kind != NODE_IDENT) return false;
```

**Why it's a real gap:**
- zercheck's free detection requires `callee->kind == NODE_IDENT`
- Function pointer calls have `callee->kind == NODE_UNARY` (deref) or other
- Pattern that fails:
  ```zer
  *(*opaque) -> void cleanup_fn = some_destructor;
  cleanup_fn(handle);    // zercheck doesn't see this as free
  use(handle);           // SILENT UAF — handle still tracked as ALIVE
  ```
- Common in plugin/callback architectures, RTOS task cleanup, etc.

**Fix path:** Track function pointer assignments and propagate `frees_param`
through the pointer's resolved target. Or conservatively mark all function-
pointer-call args as MAYBE_FREED. ~20 hrs.

### Gap 16 — `scan_unsafe_global_access` depth-32 fallback returns false (MEDIUM)

**Codebase evidence:** `checker.c:7256`:
```c
static int _scan_depth = 0;
if (_scan_depth < 32) {
    _scan_depth++;
    bool found = scan_unsafe_global_access(...);
    _scan_depth--;
    if (found) return true;
}
return false;  // <-- silent: depth-32 functions not checked
```

**Why it's a real gap:**
- Spawn data-race detection scans transitively through callees up to depth 32
- Beyond depth 32: silently returns false → no warning, no error
- Real call chains can exceed 32 levels (handler → validator → parser → ...)
- Deep call chains hide spawn-time data races silently

**Fix path:** Replace with cycle-detection-based traversal (visited set) — no
depth limit, terminate on cycles. ~15 hrs.

### Gap 17 — FuncSummary `frees_param` requires literal `free()` call name (MEDIUM) `[CLOSED 2026-04-27]`

**Codebase evidence:** `zercheck.c:280-285` in `is_free_call`:
```c
/* Check 1: named "free" */
if (callee->ident.name_len == 4 && memcmp(callee->ident.name, "free", 4) == 0)
```
Plus a heuristic for bodyless `void func(*opaque)` (lines 282-300).

**Why it's a real gap:**
- ZER function with body that internally uses non-`free` cleanup function:
  ```zer
  void destroy(*opaque h) {
      custom_cleanup(h);  // NOT named "free" — heuristic skipped
      // h's frees_param computed by scanning body for "free" calls — finds none
  }
  ```
- `destroy()`'s summary says `frees_param[0] = false`
- Caller's handle stays ALIVE after `destroy(h)` → silent UAF if used
- Same issue with `*close(*opaque)` returning non-void

**Fix path:** Extend `is_free_call` to recognize symbol-tagged "destructor"
functions OR require `@destructor` annotation OR widen heuristic to any
`void func(*opaque/*Handle)` regardless of body. ~10 hrs.

### Gap 18 — Mutual recursion FuncSummary may not converge (LOW)

**Codebase evidence:** `zercheck.c:2380-2410` GAP fix at line 2382 mentions
"earlier passes may have computed A's summary without yet knowing B's free
behavior; a second pass with B's summary available can then refine A's."

Iterative refinement runs but only ONCE more. If three+ functions are mutually
recursive with cross-dependencies, convergence may need 3+ passes.

**Why it's a real gap:**
- A calls B, B calls C, C calls A — all share a destructor parameter
- First pass: all summaries empty (frees_param=false everywhere)
- Second pass: refines once
- Third dependency may need a third refinement that never happens

**Fix path:** Loop refinement until fixed point reached or N iterations →
~5 hrs.

### Gap 34 — `IR_INDEX_READ` lacks `emit_auto_guards` — MMIO variable index silent miscompile (HIGH) **(VERIFIED)** `[CLOSED 2026-04-27]`

**Codebase evidence:** `emitter.c:9633` `emit_auto_guards` trigger list is
`IR_ASSIGN/IR_CALL/IR_RETURN/IR_INTRINSIC/IR_CALL_DECOMP`. `IR_INDEX_READ`
is **missing** from that list.

**Why it's a real silent gap:**
- Array indexing `arr[i]` (`TYPE_ARRAY`) lowers as `IR_ASSIGN` → trigger fires → `_zer_bounds_check` emitted
- Pointer indexing `reg[i]` on `volatile *u32 reg` (`TYPE_POINTER`) lowers as `IR_INDEX_READ` → trigger NOT in list → emitted C is raw `_zer_t1 = reg[i];` with NO guard
- Compiler emits warning **"MMIO index 'i' not proven in range — auto-guard inserted"** but the auto-guard was never inserted
- Hosted: SIGSEGV catches because address unmapped
- Baremetal: NO MMU — silent corruption (entire address space valid)

**Empirical verification (this session):**
```
$ ./zerc audit3_mmio.zer --emit-c -o /tmp/m.c
warning: MMIO index 'i' not proven in range (max 7) — auto-guard inserted
$ grep "reg\[i\]" /tmp/m.c
    _zer_t1 = reg[i];        # NO _zer_bounds_check before it
```

**Why the warning lies:** the warning string is hardcoded to claim insertion,
but the actual emission path doesn't run `emit_auto_guards`.

**Fix (~3 lines):** Add `IR_INDEX_READ` to the trigger list at `emitter.c:9633`,
OR call `emit_auto_guards(e, ins->expr)` inline at the top of `IR_INDEX_READ`
handler before the access.

### Gap 35 — `keep_checks[8]` / `ld_nodes[8]` fixed-buffer overflow at depth 8 (HIGH) **(VERIFIED)** `[REPRO LANDED in RO2mv]`

**Codebase evidence:** `checker.c:4347` `Node *keep_checks[8]`, `checker.c:4404`
`Node *ld_nodes[8]`.

**Why it's a real silent gap:**
- 8-deep `orelse` chain with `&local` at the end → silently bypasses `keep` parameter escape check
- `take_keep(o0 orelse o1 orelse ... orelse o7 orelse &t)` — local pointer `&t` escapes to a `keep` global
- `n=2..7`: rejected with "cannot escape local"
- `n=8`: **silently accepted** — UAF after function returns

**Empirical verification (this session):** `n=2` rejected, `n=8` produces clean
binary that runs without error.

**Why this exists:** Same anti-pattern as BUG-492 (`covered_ids[64]`). CLAUDE.md
Rule #7 explicitly prohibits fixed-size buffers for dynamic data: "Use stack-
first dynamic pattern". The keep-check code violates the project's own rule.

**Fix (~10 lines):** Stack-first dynamic with arena overflow doubling — same
pattern as `parser.c:2908`.

### Gap 36 — Shared struct read in `if`/`while`/`for`/`switch` condition is unlocked (HIGH) (agent finding) `[CLOSED 2026-04-28 — Stage 2]`

**Codebase evidence:** `ir_lower.c:1050-1062` `find_shared_root_in_stmt_ir`
inspects `NODE_EXPR_STMT` / `NODE_VAR_DECL` / `NODE_RETURN` only.
`NODE_IF/WHILE/FOR/SWITCH` lower their `cond` via `lower_expr` with no
`IR_LOCK` wrapper.

**Why it's a real silent gap:**
- `shared struct G { u32 v; } g;`
- `if (g.v > 0) { ... }` → reads `g.v` with NO `pthread_mutex_lock`
- Real cross-thread data race
- Inside async: lock-free read before `yield` → torn read possible

**Worse case:** `while (g.v < 100) { ... }` re-reads `g.v` every iteration with no synchronization. Silent infinite loop or premature exit.

**Fix:** In each control-flow case, lower `cond` through
`find_shared_root_expr`; emit `IR_LOCK` before, `IR_UNLOCK` after.

### Gap 38 — Function-return Handle bypasses zercheck_ir tracking (HIGH) **(VERIFIED on branch)** `[REPRO LANDED in RO2mv]`

**Codebase evidence:** `zercheck_ir.c:685` `ir_unwrap_alloc_expr` only
registers handles when RHS is a known alloc method (`pool.alloc`,
`slab.alloc`, etc.).

**Why it's a real silent gap:**
```zer
?Handle(Task) get_handle() { return heap.alloc(); }

?Handle(Task) mh = get_handle();
Handle(Task) a = mh orelse return;
heap.free(a);
heap.free(a);   // double-free, NOT detected
```

- Same-function double-free: caught by zercheck_ir
- Function-return double-free: silent — `dest_local` not registered as ALIVE
  because RHS is `CALL` not direct alloc method
- Runtime trap may fire IF gen counter happens to mismatch — but if Handle
  came from a fresh allocator, second free is silent UB

**Fix:** In `IR_ASSIGN`/`IR_CALL` when callee return type is `Handle/?Handle`,
register `dest_local` as ALIVE with fresh `alloc_id` (color UNKNOWN). Or
wire `FuncSummary returns_alloc[ret]` flag.

### Gap 39 — `arena.reset()` in scoped `defer` block bypasses warning + silent UAF (HIGH) **(VERIFIED)** `[PARTIAL — classification done 2026-04-27, slice-alias propagation deferred to Stage 6]`

**Codebase evidence:** `zercheck_ir.c:664` `ir_classify_method_call` —
no `IRMC_ARENA_RESET` classification. Bare `ar.reset()` triggers a soft
warning; wrapping in `defer` inside a scoped block bypasses the warning AND
runtime UAF still occurs.

**Why it's a real silent gap:**
```zer
?[*]Cell ms = ar.alloc_slice(Cell, 4);
[*]Cell s = ms orelse return;
s[0].marker = 0xDEAD;

{
    defer ar.reset();   // bypasses bare-reset warning
}

?[*]Cell ms2 = ar.alloc_slice(Cell, 8);
[*]Cell t = ms2 orelse return;
t[0].marker = 0xCAFE;

if (s[0].marker == 0xCAFE) { return 1; }   // SILENT UAF — observed at runtime
```

The warning's own suggested workaround (`defer`) creates the silent UAF.

**Fix:** Classify `Arena.reset()` as `IRMC_ARENA_RESET` in
`zercheck_ir.c:664`. At `IR_CALL` handler walk handles where
`source_color == ZC_COLOR_ARENA` and same arena symbol → mark FREED.

### Gap 40 — `switch` on `?T` accepts `.has_value` as enum variant, escapes to GCC (MEDIUM) (agent finding) `[CLOSED 2026-04-27]`

**Codebase evidence:** Checker accepts `switch (v) { .has_value => {} }` for
`?u32 v` — there's no rejection at NODE_SWITCH typing site.

**Why it's a silent gap (in checker terms):**
- ZER user thinks `.has_value` is a meaningful variant (it IS an internal
  field of `?T` representation)
- Compiles through ZER → emitted C fails: `'has_value' undeclared`
- User gets confusing GCC error from emitted code, not clear ZER error
- Not silent at runtime (compile fails downstream), but silent in the
  diagnostic sense — user can't tell what's wrong

**Fix:** At NODE_SWITCH typing site reject `TYPE_OPTIONAL` with
"use `if (v) |x|` to unwrap".

### Gap 41 — Pointer-deref-aliased free not propagated (MEDIUM) (agent) `[CLOSED 2026-04-28 — Stage 2]`

**Codebase evidence:** `zercheck_ir.c:1972` `ir_extract_compound_key` doesn't
follow `*p` to source param.

**Why it's a real silent gap:**
```zer
void sneak_free(*Handle(Task) p) {
    Handle(Task) k = *p;
    heap.free(k);
}

void main() {
    Handle(Task) h = ...;
    sneak_free(&h);  // h is freed via *p alias
    use(h);          // SILENT UAF — `frees_param[i]` doesn't fire
                     // FuncSummary only fires for direct `heap.free(p)`
                     // where p is the param itself
}
```

Caller's `h` reported as **leaked** (zercheck thinks it was never freed)
instead of UAF.

**Fix:** Extend `ir_extract_compound_key` resolution to follow `*p` to
source param. Same root as Gap 15 (function pointer free) but different
aliasing pattern.

### Gap 42 — Container-field move not tracked (MEDIUM) (agent + verified pattern in `gap5_container_move`) `[CLOSED 2026-04-28 — Stage 2]`

**Codebase evidence:** `zercheck_ir.c:1915` move-on-call-arg only handles
`NODE_IDENT` roots.

**Why it's a real silent gap:**
```zer
move struct Tok { u32 k; }
struct Box { Tok item; }

void consume(Tok t) { }

void main() {
    Box b;
    b.item.k = 1;
```

### Gap 45 — Bare-metal: shared struct `pthread_*` emission silently stripped under `__STDC_HOSTED__` (HIGH bare-metal)

### Gap 46 — Bare-metal: null pointer deref reads vector table silently (MEDIUM bare-metal)

**Why it's a real bare-metal gap:**

**Fix:** Document that bare-metal targets MUST configure MPU to mark

### Gap 26 — `_zer_shl`/`_zer_shr` shift safety bypass on negative count (HIGH) `[CLOSED 2026-04-27]`

**Codebase evidence:** `emitter.c:4505-4510`:
```c
#define _zer_shl(a, b) ({ __typeof__(b) _b = (b); \
    (_b >= (int)(sizeof(a) * 8)) ? (__typeof__(a))0 : (a) << _b; })
```

**Why it's a real silent gap:**
- Macro guards against `b >= width` (returns 0)
- Does NOT guard against `b < 0` (negative shift count)
- If `b` is a signed type holding a negative value:
  - `_b = -1`
  - `_b >= 32` is FALSE (signed comparison)
  - Falls through to `(a) << -1` → C undefined behavior
- Optimizer can produce ANY result for negative shift, including silent miscompile

**Real-world trigger:**
```zer
i32 shift_amount = compute_offset();  // could be negative
u32 result = value << shift_amount;   // _zer_shl(value, -3) = UB
```

**Fix path:** Add `(_b < 0)` to guard condition: `(_b < 0 || _b >= width) ? 0 : ...`. ~1 hr.

### Gap 27 — `@cstr` to `*u8` raw pointer destination — no bounds check (HIGH)

**Codebase evidence:** `emitter.c:2785-2820` `@cstr` emission:
```c

    /* bounds check against array.size */
} else if (dest_is_slice) {
    /* bounds check against cd.len */
} else {
    emit(e, "; ");   // <-- NO bounds check for raw *u8 destination
}
```

**Why it's a real silent gap:**
- Three destination shapes for `@cstr(dest, src)`:
  1. Fixed array `u8[256] buf` → compile-time + runtime bounds check
  2. Slice `[]u8 buf` → runtime bounds check against `.len`
  3. **Raw pointer `*u8 ptr`** → NO BOUNDS CHECK
- For `*u8` destination: emitter just does `memcpy(ptr, src.ptr, src.len)` plus null terminator
- If `src.len` exceeds the actual buffer behind `ptr`, silent buffer overflow
- Common pattern: `@cstr(some_alloc_ptr, runtime_string)` where allocation is exactly N bytes

**Fix path:** Either reject `*u8` destination at checker (require explicit slice/array)
OR add a `len` parameter to the `*u8` form: `@cstr(ptr, max_len, src)`. ~5 hrs.

## Audit method note for Gaps 20-25 (zercheck_ir.c family)

## Tier 4: FUNDAMENTAL (out of language scope)

Not gaps in ZER — fundamental properties of any safe language:

- **TOCTOU at filesystem/network level** — external state changes during
  execution
- **I/O behavior** — file content, network response, user input
- **Hardware silicon bugs** — Spectre, Meltdown, Rowhammer, cosmic rays
- **DMA buffer cache coherency** — CPU caches may have stale copies of DMA-
  written memory; requires explicit cache flush/invalidate (ZER provides
  `@cache_flush_range`/`@cache_invalidate_range` intrinsics)
- **Memory-mapped I/O ordering vs CPU memory ordering** — on weakly-ordered
  ARM Cortex-A, MMIO reads can reorder with normal memory writes; user must
  use `@barrier()` between MMIO and memory; ZER doesn't auto-insert
- **False sharing** — two unrelated atomic fields in same shared struct →
  silent perf regression (not safety)
- **Endianness assumptions** — ZER doesn't have endian-explicit types; native
  layout assumed
- **Wall-clock timing** — deadlines, jitter (OS scheduler decides at runtime)
- **Reentrant interrupts** — hardware reentrancy decision

ZER does not claim to address any of these. Same scope as Rust, Ada, Zig.

## Why do these bugs exist? — Meta-analysis (CRITICAL READING)

**Honest categorization of all 47 silent gaps by root cause:**

| Category | Count | % | What it means |
|---|---|---|---|
| **Walker/visitor incomplete** | 18 | 38% | Analysis walker handles some NODE_/IR_ kinds but misses others. Architecture-level issue. |
| **Self-discipline failures (Rule #7)** | 5 | 11% | Fixed-size buffers used for dynamic data — directly violates CLAUDE.md project policy. Code-level. |
| **Hosted-vs-baremetal split** | 7 | 15% | Runtime fallbacks designed hosted-first; baremetal got partial/silent fallbacks. Architecture-level. |
| **Cross-model boundary** | 6 | 13% | Bug spans 2+ of the 4 safety models; gap at the seam. Architecture-level. |
| **Migration debt (Phase F→G)** | 1 | 2% | Transitional state — IR-side checks computed but suppressed pending Phase G flip. Temporary. |
| **Genuine missed combination** | 5 | 11% | Untested feature × feature interaction (e.g., `threadlocal × shared`, `comptime if(0) × stack analyzer`). |
| **Heuristic limitation** | 5 | 11% | Heuristic-based detection (`is_free_call`, "bodyless void *opaque") catches the common case, misses variants. |

### So is the architecture flawed? **No.** Here's the breakdown:

**The 4-model architecture is sound.** State machines, point properties,
function summaries, and static annotations are genuinely orthogonal — no bug
in this list requires re-architecting the model decomposition. Adding a 5th
model isn't needed; existing models can absorb every fix listed.

**The architecture's INVARIANTS aren't enforced uniformly.** This is the real
issue. The 4-model design has implicit obligations:

1. **Every walker must be exhaustive.** A walker that visits NODE_IDENT but
   not NODE_FIELD is silently incomplete. Gaps 9 (range-for), 28 (nested
   move struct), 29 (struct param Handle), 36 (shared in cond), 42 (container
   field move) — all the same root pattern: walker handles some root types
   but not others.

2. **Every model boundary must compose safely.** When a check spans Models
   1+3 (e.g., function-return Handle), if Model 3 (FuncSummary) doesn't
   communicate to Model 1 (state machine) that the return is a fresh
   allocation, the state machine never registers the handle. Gap 38 is
   exactly this seam.

3. **Hosted vs freestanding is a runtime concern, not a model.** The
   hosted-first design left freestanding fallbacks ad-hoc. BM1-BM5 + Gap 9
   + Gap 45 are all "runtime fallback under `__STDC_HOSTED__ == 0` doesn't
   match hosted semantics."

4. **Self-discipline (Rule #7) is enforced by humans, not the compiler.**
   Gaps 35 (keep_checks[8]), B2 (ld_nodes[8]), Tier 4 (build_expr_key 256/64)
   are pure "team didn't follow its own rule."

### Bug origin — concrete examples per category

**Walker/visitor incomplete (38%) — the dominant cause:**
- Gap 9 (range-for `__ri`): parser desugaring uses literal "__ri" without checking reserve prefix
- Gap 28 (nested move struct): `contains_move_struct_field` doesn't recurse
- Gap 29 (struct param Handle): BUG-385 scan only one level
- Gap 36 (shared in cond): `find_shared_root_in_stmt_ir` ignores NODE_IF/WHILE/FOR/SWITCH
- Gap 42 (container field move): walker only handles bare-NODE_IDENT root
- Gap 44 (stack-depth on `comptime if(0)`): walker doesn't filter dead branches

**Self-discipline failures (11%) — most embarrassing:**
- Gap 35 keep_checks[8] / ld_nodes[8]
- Tier 4 build_expr_key stack_buf[512] / stack_buf[256] / stack[64]
- These violate CLAUDE.md Rule #7 verbatim. Project policy was set, code
  ignored it.

**Cross-model boundary (13%) — hardest to spot:**
- Gap 38 (function-return Handle): Model 3 (summary) → Model 1 (state machine) handoff missing
- Gap 39 (arena.reset in defer): Model 3 (alloc classification) → Model 1 (handle state) handoff missing
- Gap 41 (deref-aliased free): Model 3 follow-through to root param missing

**Hosted vs baremetal (15%) — design philosophy gap:**
- Gap 1 (stack overflow), Gap 9 (@probe), Gap 10 (@critical x86), Gap 11 (FPU in ISR)
- Gap 12 (packed MMIO), Gap 45 (shared struct stripped), Gap 46 (null deref vector table)

### TL;DR

**These bugs exist NOT because the architecture is wrong, but because:**

1. **Architectural invariants weren't enforced mechanically** — every walker
   is hand-written; nothing forces exhaustive node-kind coverage. A linter
   or codegen-based exhaustive switch (like the GCC `-Wswitch` strategy
   ZER already uses for 7 walkers) would catch these.

2. **Self-discipline rules need automation** — Rule #7 (no fixed buffers)
   is documented but not enforced by tooling. A grep linter or dedicated
   static analysis pass on `\[[0-9]+\]` patterns would catch new violations
   as they're introduced.

3. **Hosted-first runtime design needs an explicit baremetal contract** —
   every `#if __STDC_HOSTED__` branch should have a documented baremetal
   semantic (or a clean reject).

4. **Migration states are inherently risky** — Phase F→G dual-run leaves
   correct IR-side checks suppressed. Migration completion needs a hard
   cutover, not perpetually-in-progress.

**Most fixable category:** walker/visitor incompleteness (38% of bugs).
Each fix is ~5-15 hrs. Mechanical enforcement (exhaustive switches with
no `default:`) would prevent this entire class going forward.

**Hardest category:** cross-model boundary (13%). Each requires careful
thought about how two analyses should communicate. No mechanical fix.

**Most embarrassing category:** self-discipline (11%). Pure code-quality
issue — known rule, ignored. Trivial fix per site, but the pattern repeats.

### The brutal honest summary

ZER's safety architecture (4 orthogonal models, 30+ tracked properties)
is genuinely innovative and the right design. But:

- **The architecture is more rigorous than the implementation** — design is
  10/10, code follow-through is 7/10
- **The "100% language-safe" claim assumes implementation matches architecture**
  — currently implementation has ~47 known holes, mostly small
- **Most holes are 1-walker-fix away** from being closed — not requiring
  redesign, just exhaustive coverage
- **The team needs mechanical enforcement of architectural invariants** —
  exhaustive switches, no-fixed-buffer linter, baremetal contract tests

This is a **typical maturation curve** for a young compiler: design first,
then close gaps as they're found. The audit-driven approach (5 audit rounds
in this doc + 5 parallel agents on the branch) is exactly the right way to
mature it. The pace at which gaps are being CLOSED (e.g., the 7 gaps fixed
in `claude/cool-johnson-tNGWB`, the 4 reproducers landed in `claude/cool-johnson-RO2mv`)
is faster than the pace at which they're being FOUND, which is the right
direction.

**This document is not "ZER is broken."** It's "ZER's architecture is sound;
here are 47 implementation gaps to close to match the architecture." Closing
all 47 = ~281 hrs. ZER will then genuinely match its claim.

**Round 6** (empirical validation, this session): wrote ZER reproducer programs
inside Docker, ran through zerc, recorded exit codes and behavior. **Confirmed

"Validation Results" section below.

**Round 5** (~5000 lines: parser.c, ir_lower.c, zercheck.c move + struct scan,
emitter.c shift macros + @cstr emission): 8 more silent gaps found, all
distinct from prior rounds. Highlights:
- `_zer_shl/_zer_shr` macros don't guard `b < 0` → negative shift = UB (Gap 26, HIGH)
- `@cstr` to raw `*u8` destination has NO bounds check (Gap 27, HIGH)

- Comptime call depth=16 silently propagates CONST_EVAL_FAIL (Gap 33, LOW)

Round-5 audit method: combined principle-search (e.g. "any depth limit?",
"any one-level scan that should be recursive?", "any macro that misses negative
input?") with code-pattern scanning. Each finding verified by reading
surrounding context and tracing through to user-visible behavior.

Methodology was principle-first — search for code patterns that COULD silently
miss bugs (depth limits, fallback paths, heuristic-only matchers, conditional
compilation guards), then verify each finding by reading the surrounding
context. No agent delegation — direct grep/read work to maintain audit trust.

## STAGE 3 — Fixed-buffer cleanup + linter (~15 hrs, 4 sites) — **DISCIPLINE FOUNDATION #2**

### Approach

CLAUDE.md Rule #7: "Never use fixed-size buffers for dynamic data." This
rule is documented but not enforced — 4+ violations exist.

### Sites to convert

| Gap | Buffer | File | Hours |
|---|---|---|---|
| **35** | `keep_checks[8]` + `ld_nodes[8]` | `checker.c:4347, 4404` | 10 |
| **Tier 4** | `build_expr_key stack_buf[512]` | `checker.c:482` | 2 |
| **Tier 4** | `zercheck_ir.c stack_buf[256]` | `zercheck_ir.c:540` | 1 |
| **Tier 4** | `zercheck_ir.c Node *stack[64]` | `zercheck_ir.c:2954` | 2 |

**Conversion pattern** (proven via parser.c:2908 + BUG-492 fix):
```c
/* Before */
Node *items[8];
int count = 0;
items[count++] = node;  // silent overflow at 9

/* After */
Node *stack[8];
Node **items = stack;
int count = 0, cap = 8;
if (count >= cap) {
    cap *= 2;
    Node **new_items = (Node **)arena_alloc(arena, cap * sizeof(Node*));
    memcpy(new_items, items, count * sizeof(Node*));
    items = new_items;
}
items[count++] = node;
```

### Add CI linter

Create `tools/audit_fixed_buffers.sh` that:
- Greps `\[[0-9]+\]` patterns in walker / key-building / collection code
- Whitelist: dimension constants in struct definitions (e.g., `u8[256]` user code)
- Flags suspicious patterns in compiler internals
- Runs as `make check-fixed-buffers`; fails CI on new violations
