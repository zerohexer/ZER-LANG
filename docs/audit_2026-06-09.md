# Audit 2026-06-09 — Silent Gap Sweep

## Scope

Codebase audit focused on the IR-only safety surface (zercheck.c
ignored — production analyzer is zercheck_ir.c). Files heavily read:

- `zercheck_ir.c` (4412 lines) — production CFG-based safety analyzer
- `ir_lower.c` (3252 lines) — AST→IR lowering

Three parallel exploration agents fanned out across the largest files

findings that did not reproduce were discarded (the checker.c agent

verified reproducers; treat agent summaries as leads, not conclusions).

## Confirmed silent gaps

Eight gaps reproduced end-to-end (compile clean + observable bad
behavior at runtime, OR loud emission failure with a GCC error
where the ZER frontend should have rejected). Reproducers in
`tests/audit_2026-06-09/` with one-line file-pointer indices in the
`README.md`.

### GAP-1 — `@ptrcast` between unrelated concrete types is silent type confusion (HIGH)

**Symptom:** `*A pa = &a; *B pb = @ptrcast(*B, pa);` where `A` and `B` are
two unrelated concrete structs compiles cleanly and reads memory of
`A` as `B` at runtime. No compile-time error, no runtime trap.

**Root cause:** `checker.c:6262` only triggers the provenance check
when the SOURCE is `*opaque` (`eff->pointer.inner->kind == TYPE_OPAQUE`).
For concrete-to-concrete `@ptrcast`, the source is not opaque, the
provenance branch is skipped, and the cast falls through to a plain
`(B*)pa` in emitted C.

**Why this is a real loss vs documented contract:** Both CLAUDE.md
and the section docstring for the intrinsic say `@ptrcast` is
"provenance-tracked." Users reading the doc will reasonably expect
the same protection `@pun` provides (which DOES insert a runtime
`_zer_opaque.type_id` mismatch trap). `@pun` traps;
`@ptrcast` between concrete types does not.

**Fix sketch:** When both source and target are concrete (non-opaque,
non-`*opaque`) pointer-to-different-types, either:
  - reject at compile time with a hint pointing the user at `@pun`
    (which IS the audit-visible escape hatch for raw casts), or
  - emit the same runtime type-id check `@pun` emits.

The first option matches the "explicit-intent primitives" philosophy
in CLAUDE.md's Definition A architecture — the user's intent (raw
pun vs validated provenance round-trip) should be visible at the use
site.

**Reproducer:** `tests/audit_2026-06-09/gap_ptrcast_concrete_unrelated.zer`.

### GAP-2 — `--no-strict-mmio` drops the runtime range+alignment check (HIGH)

**Symptom:** With `--no-strict-mmio`, variable-address `@inttoptr`
emits the raw cast `((uint32_t*)(uintptr_t)(addr))` with NO range
check and NO alignment check. The flag is documented as
"allows `@inttoptr` without `mmio` declarations" — i.e., relaxes
COMPILE-TIME strictness — but it also strips the RUNTIME trap.

**Verification:** `zerc this.zer --emit-c -o a.c` emits
`_zer_trap("@inttoptr: address outside mmio range", ...)` and
`_zer_trap("@inttoptr: unaligned address", ...)` inline at the
inttoptr expression. Adding `--no-strict-mmio` strips both checks.

**Why it matters on bare-metal:** A `--no-strict-mmio` build pointing
at a runtime-computed address can silently write to any peripheral
region. Misaligned writes BusFault on Cortex-M0/M0+ at runtime;
out-of-range writes corrupt unrelated peripherals.

**Fix sketch:** The `--no-strict-mmio` flag should only relax the
compile-time enforcement (the requirement that `@inttoptr` be inside
a declared `mmio` range). The emitted runtime range+align check
should be unconditional whenever the user HAS declared ranges, and
the alignment check should be unconditional always.

**Reproducer:** `tests/audit_2026-06-09/gap_nostrict_mmio_drops_runtime.zer`.

### GAP-3 — `alloc_ptr` global alias UAF is silent at both gates (HIGH)

**Symptom:** Storing a `*T` from `heap.alloc_ptr()` in a global
optional pointer (`?*T g_ptr`), freeing the local, then later
dereferencing the unwrapped global compiles clean AND runs through
without trap.

**Root cause stack:**
1. `g_ptr = p` triggers the escape check, which marks `p.escaped = true`
   to suppress the leak warning (correct behavior for the local).
2. The analyzer does NOT register the global field `g_ptr` as a
   compound key tracking `p`'s alloc_id (the same gap class as
   audit_2026-06-02 Agent 2 #2 — Handle variant was added later,
   `*T` was not).
3. Subsequent `g_ptr orelse return` creates a fresh local with no
   tracked state. zercheck_ir treats it as ALIVE.
4. `_zer_slab_free_ptr` does not maintain a per-slot generation
   counter on `*T` (unlike Handle), so the dereference of the
   freed pointer reads stale slab memory without trapping.

**Why this is worse than the Handle variant:** the Handle global-alias
UAF (audit_2026-06-02 Agent 2 #2) is caught at runtime by the slab
gen counter — compile-time still misses it, but runtime catches it.
The `*T` variant has no such safety net. CLAUDE.md's claim about
`alloc_ptr` ("100% compile-time safe for pure ZER code") is the
documentation discrepancy — the audit confirms it's aspirational.

**Fix sketch:** Either
  - Register globals storing `*T` from `alloc_ptr` as compound keys
    in zercheck_ir, so the subsequent unwrap's deref is treated as
    accessing the same alloc_id as the local `p` (compile-time
    detection), OR
  - Add a per-slot generation counter to `*T` (turning it into
    Handle-with-different-syntax — undesirable, defeats the
    `alloc_ptr` design).

The first option is the right path.

**Reproducer:** `tests/audit_2026-06-09/gap_alloc_ptr_global_alias_uaf.zer`.

### GAP-4 — Function-pointer free is not tracked → silent double-free (HIGH)

**Symptom:** Calling `fp(h)` through a `*(Handle(T)) -> void` function
pointer that the target body frees the handle. zercheck_ir does not
propagate the free across the indirect call. A subsequent direct
`heap.free(h)` is the second free from a runtime perspective, but
neither compile-time nor runtime catches it. `_zer_slab_free` is
intentionally lenient on already-freed handles (no `h_gen != gen[idx]`
trap on free, only on get).

**Status:** documented in `docs/limitations.md` as "Gap 15 reverified"
(2026-05-25) — known but deferred because resolving requires either
function-pointer target-set analysis or a conservative "any indirect
call invalidates tracked handles" model. **The audit confirms the
end-to-end silence (exit=0 on the double-free).**

**Fix sketch:** Conservative "any indirect call may free any handle"
model — turn the indirect call into a barrier that widens every
ALIVE handle to MAYBE_FREED. Over-rejects, but matches Rust's
treatment of `unsafe fn` calls. Less intrusive: only widen handles
of types that appear in the function pointer's signature.

**Reproducer:** `tests/audit_2026-06-09/gap_funcptr_double_free.zer`.

### GAP-5 — orelse-reassignment overwrite leak (MEDIUM)

**Symptom:** `Handle(T) h = pool.alloc() orelse return;` followed by
`h = pool.alloc() orelse return;` leaks the first allocation silently.
IR_COPY's overwrite-leak check (F3.1 partial fix 2026-05-03) covers
the bare `h = pool.alloc()` form but not the orelse-decomposed shape.

**Status:** documented in audit_2026-06-02 Agent 2 #1 as "still open."
The audit confirms it still reproduces in the 2026-06-09 codebase.

**Fix sketch:** In `zercheck_ir.c` IR_ASSIGN IRMC_NONE branch
(handling orelse-decomp into a Handle dest local), when dest is
currently ALIVE and the assignment isn't a free-call, emit the
overwrite-leak diagnostic the same way IR_COPY does for the bare
case.

**Reproducer:** `tests/audit_2026-06-09/gap_orelse_overwrite_leak.zer`.

### GAP-6 — Array element double-free with variable index (MEDIUM)

**Symptom:** `heap.free(arr[k])` followed by `heap.free(arr[0])` where
`k` was 0 (post-VRP, runtime-constant) compiles clean and runs
silently. `ir_extract_compound_key` only accepts `NODE_INT_LIT`
indices, so `arr[k]` doesn't match the same compound key as `arr[0]`.

**Status:** known design choice (compound-key indexing limited to
literal indices to keep the analyzer fast and conservative). Runtime
saves the most-common case via slab gen mismatch — but only on
subsequent `get`, not on the second `free`. The double-free itself
is silent.

**Fix sketch:** Extend `ir_extract_compound_key` to accept indices
proven const by VRP, OR widen the compound-key match to "any index"
(over-rejects, but matches conservative Rust treatment).

**Reproducer:** `tests/audit_2026-06-09/gap_arr_var_index_dfree.zer`.

### GAP-7 — Container monomorphization with composite type args (MEDIUM, UX)

**Symptom:** `Box(?u32)` emits `struct Box_?u32 { ... }`, `Box(*u32)`
emits `struct Box_*u32`, `Pair(Handle(Item))` emits
`struct Pair_Handle(Item)`. All produce GCC syntax errors. The error
is "loud" at the GCC stage but the diagnostic points the user at
emitted C, not the ZER source where the container instantiation lives.

**Status:** noted in audit_2026-06-02 Agent 4 A1 as still open.

**Fix sketch:**
  - Reject non-identifier type args at the checker with a clean
    ZER error (matches Zig's behavior — non-identifier generic args
    are a syntax error), OR
  - Emit a name-mangling pass: `?` → `_opt_`, `*` → `_ptr_`,
    `[*]` → `_slice_`, `(` → `_lp_`, `)` → `_rp_`. Must be
    reversible for type equality.

**Reproducer:** `tests/audit_2026-06-09/gap_container_ptr_optional_arg.zer`.

### GAP-8 — Arena escape via struct copy through function parameter (MEDIUM)

**Symptom:** Direct `c.ptr = p` where `p` is arena-derived is caught
(checker.c:3879+). The same value laundered via:

```zer
local.ptr = p;      // local.ptr inherits arena-derived
take(local);        // value-typed Container param ct = copy of local
                    // arena-derived flag does NOT propagate through copy
void take(Container ct) { c = ct; }  // store copy to global
```

slips through the static check. Runtime catches it via the
libc-malloc-wrap fault when `c.ptr` is later dereferenced (on
hosted) — but with a generic "memory access fault" trap, not the
documented "arena-derived pointer escape" compile error. On
bare-metal without malloc-wrap, fully silent.

**Fix sketch:** Two complementary changes:
1. Propagate `is_arena_derived` from a field assignment up to the
   containing struct local (`local.ptr = p` → `local.is_arena_derived = true`).
2. Propagate `is_arena_derived` through value-typed struct parameters
   into the callee's parameter symbol — so `ct.ptr` in `take` carries
   the arena-derived flag and the assignment `c = ct` is rejected
   by the existing global-store check.

The two together close the launder for the common
"store-into-field-then-pass-by-value" pattern. Doesn't require any
new tracking infrastructure — just two extra propagation sites.

**Reproducer:** `tests/audit_2026-06-09/gap_arena_escape_via_struct_copy.zer`.

## Technical debt observations

### Dead IR opcodes — defensively trapping, but design unresolved

`ir.h` still declares 14 opcodes that `ir_lower.c` never emits
(`IR_POOL_ALLOC`, `IR_SLAB_ALLOC`, `IR_FIELD_WRITE`, `IR_INDEX_WRITE`,
`IR_ADDR_OF`, `IR_DEREF_READ`, `IR_CALL_DECOMP`, `IR_INTRINSIC_DECOMP`,
`IR_ORELSE_DECOMP`, `IR_SLICE_READ`, the various decomposed
allocator op variants, and IR_SPAWN). The emitter previously emitted
`/* 3AC op N — TODO */` comments which would have silently
miscompiled if any of these ever started being emitted; that gap was
closed (around audit-2026-05-08) by replacing the stubs with
`fprintf(stderr) + _zer_trap("not implemented")`, making any
regression LOUD at runtime. Good defense.

The opcodes remain in the enum because the original design intended
a fuller 3AC decomposition that wasn't taken. **Recommended cleanup:**
either delete the enum values + handlers (~200 LOC removal) or wire
`ir_validate` to reject these as malformed IR until/unless a future
decomposition pass produces them.

This is the same finding as audit_2026-06-02; calling it out again
because the cleanup hasn't happened.

### Two parallel safety-wrapper paths in emitter

Most safety wrappers (`_zer_bounds_check`, `_zer_trap` for div-overflow,
`_zer_shl`/`_zer_shr` for shift width) are now mirrored between
`emit_expr` (AST path, legacy) and the IR-path handlers
(`emit_rewritten_node` for compound assigns, IR_BINOP for binary).

The IR migration deleted AST function-body emission in 2026-04-19,
so the AST `emit_expr` is mostly dead code today — but it remains
in `emitter.c` as a fallback for some statement-expression contexts.
Each safety wrapper must be maintained in both locations. Two known
sites (compound `/=` and `%=`) still need the INT_MIN signed-overflow
trap on the AST side, per `docs/limitations.md` "AST emit_expr
compound /= and %=." Not a silent gap today, but a documented
defense-in-depth gap that will become a real gap if function bodies
ever route through `emit_expr` again.

**Recommended cleanup:** delete dead `emit_expr` branches that are
no longer reached from function bodies, OR document each
remaining-reachable branch with the specific caller, so future
sessions know what's load-bearing.

### `--no-strict-mmio` flag semantics underspecified

Per CLAUDE.md the flag is "allow @inttoptr without mmio
declarations." The intent is to relax compile-time strictness; the
runtime check is documented as preserved. As GAP-2 shows, the
current implementation drops the runtime check too. Either the
documentation or the implementation is wrong; this audit recommends
aligning the implementation with the documentation (preserve
runtime check).

If the user wants ZERO checks (compile + runtime), a new flag
`--unsafe-mmio` or `--no-mmio-trap` would be a separate axis. Don't
overload one flag with two semantics.

## Methodology notes

- **Spawn-then-verify protocol works.** Three parallel agents found
  many candidates; verification with reproducers narrowed to the
  eight real gaps above. The checker.c agent in particular returned
  a "no silent gaps detected" verdict that contradicted seven of
  the eight findings. Treat agent reports as research — verify
  every claim before acting.

- **`ir_extract_compound_key` is again the recurring unifier.** Every
  silent gap with array/field tracking traces back to filtering on
  `arg->kind == NODE_IDENT` or `index->kind == NODE_INT_LIT` when
  the analyzer should walk through more shapes. Mechanical sweep:
  `grep -nE 'kind == NODE_IDENT|kind == NODE_INT_LIT' zercheck_ir.c`
  — every match is a candidate "shape" silent gap.

- **Runtime safety nets mask compile-time gaps.** The Handle gen
  counter (`_zer_slab_get`) catches many compile-time misses for the
  Handle variant; `*T` from `alloc_ptr` has no equivalent net, so
  the same shape gap is "Handle: runtime caught, *T: silent." When
  auditing, always test BOTH variants.

- **`@pun` is the right shape; `@ptrcast` between concretes is the
  gap.** Future audits of cast intrinsics should compare each
  intrinsic against the runtime check `@pun` emits — anything weaker
  is a silent type-confusion gap.
