# Codebase audit — 2026-05-02

Comprehensive audit of the ZER compiler at branch `claude/cool-johnson-goV41`
(HEAD `9d930ab`, post Stage 4 F4-F7 asm work). Six parallel deep audits ran
across IR pipeline, checker.c, emitter.c, asm Stage 4, concurrency/async,
parser/lexer + remaining roadmap gaps. Findings below are **verified against
actual code** — agent-only claims not confirmed empirically are marked
NEEDS-VERIFICATION.

This is a snapshot, not a backlog. Open new entries in `docs/limitations.md`
or `docs/4-27-2026-gaps.md` if work is scheduled. Audit reproducers landed
in `tests/zer_gaps/` (excluded from `make check`).

## Confirmed silent gaps — both compile-time AND runtime miss the bug

These are the highest-impact: the program compiles cleanly, runs without trapping,
and produces wrong behavior. On baremetal the fault never surfaces.

### G1. `@atomic_store` accepts mismatched value type — silent narrow/widen
- **Severity:** HIGH (silent type confusion; on baremetal, no runtime catch)
- **File:** `checker.c:6619-6623`
- **Reproducer:** `tests/zer_gaps/atomic_store_subtle_narrow.zer`
- **Why silent:** lines 6619-6623 only verify `arg_count != 2`. No type/width
  validation on the first arg pointer or the value arg. GCC's
  `__atomic_store_n` then silently coerces the value to the pointer's inner
  type. ZER's CLAUDE.md table claims "first arg must be `*shared T` where T
  is integer of width 1/2/4/8" — the check exists for `@atomic_add`/`@atomic_load`
  but is missing here.
- **Fix:** mirror the @atomic_add path (lines 6634-6655): unwrap pointer-inner,
  call `zer_atomic_width_valid`, and verify the value arg coerces to the
  inner type. ~10 LOC.

### G2. `@atomic_cas` accepts any types for expected/desired
- **Severity:** HIGH (same class as G1)
- **File:** `checker.c:6624-6628`
- **Reproducer:** `tests/zer_gaps/atomic_cas_no_type_check.zer` (struct as
  target → GCC catches; bool→u32 silently coerces)
- **Fix:** validate first arg is `*T` where T is integer 1/2/4/8 bytes;
  expected and desired widths must match T. ~12 LOC.

### G3. `@atomic_load` validates width inline instead of via `zer_atomic_width_valid`
- **Severity:** LOW (cosmetic — same outcome, different code path)
- **File:** `checker.c:6604-6607` uses inline `aw != 8 && aw != 16 && ...`
  whereas line 6642 uses the extracted predicate.
- **Fix:** use `zer_atomic_width_valid(aw / 8)` for parity with the
  Coq-extracted spec.

### G4. Function-pointer call to a free function bypasses zercheck
- **Severity:** HIGH (silent UAF / silent double-free)
- **File:** `zercheck_ir.c:643` (`if (callee->kind != NODE_IDENT) return false`)
- **Reproducer:** `tests/zer_gaps/funcptr_free_uaf_silent.zer`
- **Confirmed runtime:** `fn(h); tasks.free(h)` compiles with only a
  stack-depth warning. The double-free at runtime is masked because
  `_zer_slab_free` is idempotent (just `used[idx]=0; gen[idx]++`).
  But `gen` advances twice — any handle the program retained becomes
  observably stale at the wrong moment, **and** the same gap covers
  destructors that aren't idempotent (refcount decrement, file close).
- **Fix:** when funcptr target is a NODE_IDENT assigned from a known
  function, resolve through the assignment to the underlying symbol.
  Or: warn (not error) on funcptr-call-with-Handle-arg as a heuristic
  upgrade. Real fix: add a "free" attribute to function declarations.

### G5. `--probe-mode=raw` silently always returns has_value=true
- **Severity:** HIGH on baremetal (Roadmap Gap 9)
- **File:** `emitter.c` raw probe-mode emitter, ~line 4707
- **Reproducer:** `tests/zer_gaps/probe_raw_silent_success.zer`
- **Why silent:** `_zer_opt_u32` returned with has_value=1 for every read,
  including faulting addresses on baremetal (no signal handler). The
  `if (probe(addr)) |v|` pattern always takes the success branch.
  This defeats `@probe`'s contract as the **safe** MMIO read.
- **Fix:** under `--probe-mode=raw`, either reject `@probe` at compile
  time (force the user to `@inttoptr`), or emit a hardware-based probe
  per arch (DR/MMU watch). Document is currently in CLAUDE.md but
  enforcement is missing.

### G6. `@once` has no hosted/freestanding gate
- **Severity:** MEDIUM (link error or no-op on baremetal)
- **File:** `checker.c:10383-10388` (NODE_ONCE handler is body-recurse only)
- **Why silent:** `@once` body emits `pthread_mutex_*` under `__STDC_HOSTED__`
  guard. On freestanding, the once flag is set on first call but the
  body never executes thread-safely. Dual to G5: emitter has the conditional
  guard, checker does not.
- **Fix:** at NODE_ONCE, error if `c->is_freestanding` (or whatever the
  flag is). Mirror spawn's ISR check at `checker.c:10596`.

### G7. CLI argument parser: `argv[1]` is unconditionally the input file
- **Severity:** MEDIUM (UX — confusing error)
- **File:** `zerc_main.c:215` (`input_path = argv[1]`)
- **Repro:** `./zerc --probe-mode=raw foo.zer` → `error: cannot open '--probe-mode=raw'`.
  Reorder to `./zerc foo.zer --probe-mode=raw` works.
- **Fix:** scan argv for the first non-`-` arg as input_path; iterate
  flags from start. Standard argparse pattern.

## Confirmed but lower-impact silent gaps

### G8. `naked` LR/SC pairing state machine resets per asm block
- **Severity:** MEDIUM (false positives on legitimate cross-block patterns;
  also the dual: end-of-block unmatched LL never errored)
- **File:** `checker.c:9976-10105` (F7-light LR/SC state)
- **Two related issues, same root cause** (per-asm-block scope):
  1. `lr.d` in asm-block-A and `sc.d` in asm-block-B (same naked function)
     errors "SC without LR" — false positive. Real RISC-V/ARM patterns
     do this; the exclusive monitor is per-hart, not per-block.
  2. `lr.d` followed by no `sc.d` in the same block ends the block with
     `ll_pending=true` but the loop exits without erroring. Reservation
     leaks across the rest of the function; subsequent atomics on the
     same line silently fail. No diagnostic.
- **Fix:** track `ll_pending` at function scope, not loop-local. End-of-function
  check fires the unmatched-LL error. ~30 LOC.

### G9. AVX-512 high registers (xmm16-31, k0-k7) accepted by checker but rejected by GCC
- **Severity:** MEDIUM (no silent miscompile; confusing UX — checker green, GCC red)
- **File:** `src/safety/asm_register_tables_x86_64_avx512f.c` validates them;
  emitter doesn't add `__attribute__((target("avx512f")))`. Hardcoded probe
  in `scripts/gen_register_tables.sh` doesn't use `-mavx512f`, so x86_64.zerdata
  classifications without `--target-features=avx512f` reject. Outcome: green
  checker on `--target-features=avx512f`, red GCC because emitter omits the
  function attribute.
- **Fix (v1.0):** reject xmm16-31/k0-k7 in checker even with avx512f flag
  until the emitter adds the function attribute. Document deferral.
- **Fix (v1.1):** emit `__attribute__((target("avx512f")))` on functions
  whose asm references high regs. ~15 LOC in emitter.

### G10. ARM LDXR/STXR alignment classification missing in `aarch64.zerdata`
- **Severity:** MEDIUM (silent UB on misaligned ARM exclusive load)
- **File:** `arch_data/aarch64.zerdata:31-65`
- **Why silent:** ARM exclusive monitor is **not atomic on misaligned
  addresses**; the load completes but the reservation isn't set, so SC
  silently returns success=0 forever. RISC-V LR/SC entries correctly
  carry `ALIGNED(8)`/`ALIGNED(4)`. ARM entries don't.
- **Fix:** add `operand[1].constraint = ALIGNED(N)` to LDXR/LDAXR/STXR/STLXR
  entries in `aarch64.zerdata`. Run `make gen-asm-tables` to vendor.

### G11. Constraint operand binding assumes positional-output-then-input order
- **Severity:** MEDIUM (silent constraint skip on reordered operand declarations)
- **File:** `checker.c:10146-10160` (Pass A operand binding)
- **Why silent:** instruction-table operand[N] binds to N-th binding using
  GCC's "outputs first then inputs" convention. If user writes
  `outputs: { "rdx" = rem }; inputs: { "rcx" = divisor }` and the
  constraint targets the divisor (operand[0]), Pass A binds operand[0]
  to outputs[0]=rem, then either skips (wrong type) or checks the wrong
  variable. NONZERO/COMPOUND/ALIGNED/BOUNDED can be silently skipped.
- **Fix:** in Pass A, when binding count < operand count, error
  "asm requires N operand bindings but found M." When constraint
  expects integer and binding is a pointer/struct, error
  "constraint refers to unbinable operand position."

### G12. `cc` pseudo-clobber accepted on aarch64/riscv64 (not portable)
- **Severity:** LOW (cosmetic; GCC catches at emit)
- **File:** `checker.c:10373-10377`
- **Fix:** gate the `cc` skip to `c->target_arch == 1` (x86_64).

### G13. `safety:` string content never validated (only length)
- **Severity:** LOW (audit-trail degrades silently — S4 rule satisfied on paper)
- **File:** `checker.c:9516-9520`
- **Repro:** `safety: "                              "` (30 spaces) accepted.
- **Fix:** require ≥20 printable non-space chars in the safety string.

### G14. ThreadHandle has binary `joined` flag, not tristate ALIVE/MAYBE_JOINED/JOINED
- **Severity:** MEDIUM (false negatives or false positives on conditional joins)
- **File:** `zercheck_ir.c:2067, 3407-3410`
- **Why silent:** Handle tracking has MAYBE_FREED via CFG merge; ThreadHandle
  doesn't. Conditional join (`if (cond) { th.join(); }`) at function exit
  goes through CFG merge but only the linear `joined` boolean — the merge
  picks one branch's value and discards the other.
- **Fix:** mirror Handle's tristate state on IRThreadTrack. Same merge logic
  at CFG join (`zercheck_ir.c:381-460`).

### G15. Move struct in async state struct has no lifetime tracking
- **Severity:** HIGH (silent use-after-move in async)
- **File:** `emitter.c:3541-3577` (`collect_async_locals` doesn't gate on `is_move`)
- **Why silent:** state-struct field for a moved-out value remains valid
  C-wise. zercheck_ir doesn't propagate HS_TRANSFERRED to async state-struct
  fields when they get consumed across yield boundaries.
- **Fix:** in zercheck_ir, treat IR_STATE_FIELD_READ/WRITE on move-typed
  state fields the same as locals — propagate HS_TRANSFERRED state through
  the field.

### G16. Lock-across-await passes through helper-fn shared-param access
- **Severity:** MEDIUM (deadlock not caught when reading shared field via helper)
- **File:** `checker.c:5030-5040` (in_async_yield_stmt check is direct-access only)
- **Why silent:** `await helper(&shared_global)` where helper reads `s.field`
  takes the auto-lock inside helper. The `in_async_yield_stmt` flag is set
  at the `await` statement but the check fires only on direct
  NODE_FIELD/NODE_POINTER_DEREF of shared types in the current scope. Helper's
  body is not consulted.
- **Fix:** extend `FuncProps` with `accesses_shared_param[i]: bool`. At
  `await call(args)`, if any arg is a shared pointer AND callee accesses
  it under lock, error.

### G17. async + condvar: `prescan_async_temps` doesn't recurse into intrinsic args
- **Severity:** MEDIUM (latent — would emit "switch label inside statement-expr" GCC error)
- **File:** `emitter.c:3470-3511`
- **Repro pattern:** `defer { @cond_wait(sh, ?opt orelse {fb}); }` — orelse
  inside `@cond_wait`'s argument is a NODE_INTRINSIC subtree that isn't
  walked by `prescan_expr_for_orelse`.
- **Fix:** in `prescan_expr_for_orelse`, add `NODE_INTRINSIC` case that
  walks every arg.

## Mid-priority technical debt

### TD1. AST→IR migration drift — VRP-on-IR coverage smaller than AST VRP
- **File:** `vrp_ir.c` (349 lines vs `checker.c` VRP infrastructure)
- **What's missing:** range derivation through call-graph (`find_return_range`
  for IR), const symbol resolution beyond literal modulo/AND, struct-field
  init lookup. CLAUDE.md notes "VRP 100% via address_taken" — that 100%
  is on AST. IR-side coverage is intentional but documented thinly.
- **Why deferable:** dual-run validates 0 disagreements across 3143 programs;
  zercheck.c is still primary. When zercheck.c is deleted (Phase G), this
  becomes a real gap.

### TD2. Emitter positional struct initializers (CLAUDE.md violates own rule)
- **Files:** `emitter.c:1789` (Arena `over`), `:1031, 5313` (slice from string),
  `:118` (null literal), `:4668, 4701, 4706, 4713` (`_zer_probe`),
  `:5196, 5197` (auto-slab alloc).
- **Why deferable:** All emit valid C today. The fragility activates only
  if the runtime struct field order changes. Rule was added after these
  sites. Cleanup is mechanical.

### TD3. `ir_classify_method_call` (no _ex) backcompat wrapper still has callers
- **Files:** `zercheck_ir.c:818-820` defines the wrapper.
- **Roadmap Gap 32** is partially closed; sweep call sites for non-_ex
  variant and migrate. Forward-compat: deprecate the wrapper after migration.

### TD4. `ir_validate` defer-balance check is reachability-only, not depth-aware
- **File:** `ir.c:280-320`
- **Symptom:** an `IR_DEFER_PUSH` with reachable `IR_DEFER_FIRE` passes, even
  if the fire's `cond_local` (depth) doesn't actually pop the defer in
  question. Cross-CFG-path partial fires are theoretically possible.
- **Why deferable:** `ir_lower.c` is the only producer and pushes/fires
  at construction time, not from user input. No empirical regressions.

### TD5. `scan_unsafe_global_access` skips function-pointer callees
- **File:** `checker.c:7456-7474`
- **Symptom:** spawn target `void target() { handler(); }` where `handler` is
  a global funcptr — the scanner can't resolve and assumes safe. Dual of G4.
- **Fix:** when callee is `NODE_UNARY` (deref) or `NODE_FIELD` of funcptr type,
  conservatively return "may touch unsafe globals" → error.

### TD6. F7-light hardcodes LL/SC mnemonics in `checker.c` instead of using table flags
- **File:** `checker.c:9976-9995`
- **Why deferable:** works correctly today; maintenance burden is "two
  places to edit when adding a new LL/SC ISA." Schema extension to add a
  C3 sub-flag for "this mnemonic IS the LL half" is straightforward.

## Refuted agent claims

The following agent findings did NOT verify under direct test:

- **IR audit Finding 1 (compound `/=` overflow trap missing):** the AST-level
  forced-division-guard catches the divisor-not-proven-nonzero case before
  IR emission. Test `i32 x; i32 y; x /= y;` is rejected by checker (`y` not
  proven nonzero), so the IR-level compound trap is dead-code-eliminated for
  legitimate programs. The compound `/=` IR_BINOP path was previously fixed
  in BUG-612 — agent didn't see the fix.
- **IR audit Finding 5 (array bounds in IR_INDEX_READ skips):** unprovable
  array index actually emits the auto-guard (warning printed, runtime
  guard inserted, OOB silently swallowed via `return`). Agent missed the
  `emit_auto_guards` IR pass.
- **Concurrency Finding 11 (`@atomic_load` on `*opaque`):** does error
  ("argument must be pointer to integer"). Verified by repro.

## Stage 5+ recommendations

Priority order for remaining work:

1. **Atomic intrinsic type validation gaps (G1-G3)** — small surface, high
   impact, mirrors existing add/sub paths. ~30 LOC total.
2. **Funcptr-free attribution (G4 / TD5)** — start with the warning-only
   heuristic on funcptr-with-Handle-arg; promote to error after stabilizing.
3. **`@probe` and `@once` baremetal contracts (G5, G6)** — design call:
   reject at compile, or guarantee semantics across hosted/freestanding.
4. **F7-light LL/SC scope expansion (G8)** — function-wide tracking + end-of-function
   unmatched-LL error.
5. **AVX-512 high register dual-fix (G9)** — interim reject in checker;
   final fix via function attribute emission.
6. **AArch64 zerdata alignment classifications (G10)** — single zerdata edit
   + regen.
7. **Operand binding strictness (G11)** — single Pass A guard against
   reordered/short binding lists.
8. **Move struct + async state struct lifetime (G15)** — extend zercheck_ir
   state propagation to state-struct fields.
9. **Lock-across-await transitive helper detection (G16)** — FuncProps
   extension.
10. **Async + condvar prescan recursion fix (G17)** — single case in
    `prescan_expr_for_orelse`.

After 1-10 land, a follow-up audit using the same six parallel sweeps should
find ≤3 new gaps (post-convergence audit pattern from CLAUDE.md "Multi-Round
Audit Protocol").
