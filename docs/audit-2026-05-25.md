Multi-agent exploration audit + reproducer testing across the IR-based

AND runtime trap missing) and clean up technical debt left over from the
AST → IR transition.

## Fixed in this session

### Gap 48 (BUG-661) — Forced division guard bypassed by complex divisors (HIGH)

ZER's safety promise:
> Division by zero | Forced guard (compile error if divisor not proven nonzero)

The IR-path emitter (`IR_BINOP` in `emitter.c:9404`) intentionally omits
a runtime div-by-zero check on the assumption the checker has rejected
unproven divisors. The checker's check at `checker.c:2587-2618` only
catches NODE_IDENT, NODE_FIELD, NODE_CALL — every other expression kind
fell through with no else branch.

Verified silently accepting:

| Pattern | What slipped | Runtime |
|---|---|---|
| `100 / arr[i]` | NODE_INDEX divisor | SIGFPE (hosted), hardware fault (baremetal) |
| `100 / *p` | NODE_UNARY divisor | SIGFPE / fault |
| `100 / (a - b)` | NODE_BINARY divisor | SIGFPE / fault |
| `100 / @truncate(u32, x)` | NODE_INTRINSIC divisor | SIGFPE / fault |
| `x /= s.field` | compound-assign NODE_FIELD divisor (only IDENT was checked) | SIGFPE / fault |
| `x %= arr[i]` | compound-assign NODE_INDEX divisor | SIGFPE / fault |

Hosted SIGFPE *looks* like a trap but bypasses ZER's `_zer_trap` machinery
— no source line, no controlled message, just a hardware signal. Baremetal
gets the raw exception with even less context.

Fix in `checker.c`: added catch-all `else if (kind != literal)` to both
the `/` `%` binary path and the `/=` `%=` compound-assign path, plus a
special-case for `@size(T)` which is always > 0.

Tests: 6 negative tests in `tests/zer_fail/div_zero_*.zer`, 1 positive
test in `tests/zer/div_by_size_intrinsic.zer`.

### Gap 49 (BUG-662) — Overlapping `mmio` ranges silently accepted (MEDIUM)

`mmio 0x40000000..0x40000FFF; mmio 0x40000800..0x40001000;` compiled clean
pre-fix. The two ranges describe the same physical region, so `@inttoptr`
range checks pick whichever was registered first as authoritative — an
arbitrary, brittle choice that almost always indicates a copy-paste error.

Fix in `checker.c` NODE_MMIO handler: pre-registration overlap loop with
both halves of the conflict shown in the error.

Test: `tests/zer_fail/mmio_range_overlap.zer`.

### Gap 15 reverified — function-pointer free not tracked by zercheck_ir (HIGH, OPEN)

```zer
void free_task(Handle(Task) h) { g_tasks.free(h); }
void (*fp)(Handle(Task)) = free_task;
fp(h);          // does not propagate the free
return h.id;    // UAF, should error
```

Today's diagnostic is a *secondary* "handle never freed" leak warning,
not the *primary* UAF on `h.id`. Both warnings indicate that
zercheck_ir doesn't see the indirect call as freeing the handle, so its
state remains ALIVE. A leak-suppression workaround (e.g. wrapping in
defer) would silently miss the UAF.

Closing this requires resolving callees through `TYPE_FUNC_PTR`'s target
list or a conservative "any indirect call invalidates tracked handles"
model. Both materially larger than this session's scope.

### Dead IR opcodes (technical debt)

`ir.h` declares these opcodes that `ir_lower.c` NEVER emits (verified by
grep):

```
IR_POOL_ALLOC, IR_POOL_FREE, IR_POOL_GET,
IR_SLAB_ALLOC, IR_SLAB_FREE, IR_SLAB_FREE_PTR, IR_SLAB_ALLOC_PTR,
IR_ARENA_ALLOC, IR_ARENA_ALLOC_SLICE, IR_ARENA_RESET,
IR_RING_PUSH, IR_RING_POP, IR_RING_PUSH_CHECKED,
IR_SPAWN,
IR_FIELD_WRITE, IR_INDEX_WRITE,
IR_ADDR_OF, IR_DEREF_READ,
IR_CALL_DECOMP, IR_INTRINSIC_DECOMP, IR_ORELSE_DECOMP, IR_SLICE_READ,
```

Per `zercheck_ir.c:751-760` comment and `CLAUDE.md` "Phase 8d collapse":
allocator builtins flow through IR_ASSIGN / IR_CALL with AST passthrough
in `inst->expr`. The specialized opcodes are remnants of a 3AC
decomposition direction that wasn't taken.

The risk: handlers EXIST for these opcodes both in `zercheck_ir.c`
(non-trivial — e.g. IR_FIELD_WRITE at line 2866 has 65 lines of dead
logic) and in `emitter.c` (line 9677 emits literal `/* 3AC op %d —
TODO */` comment as C output). If a future refactor starts emitting any
of them, the emitter stubs silently produce TODO comments — neither
compile-time nor runtime would catch the regression.

Two cleanup paths, neither in scope this session:
1. Delete the dead enum values + handlers (smallest, ~200 LOC removal).
2. Wire `ir_validate` to reject unreachable opcodes as malformed IR,
   keeping the placeholders for documented future decomposition.

### VRP gaps in `vrp_ir.c` (FOUNDATION-phase, deferred)

- Does not invalidate range state on `@atomic_*` writes (today's atomics
  conservatively invalidate only the address-taken set, missing
  `*shared` globals tracked through a chain).
- Does not propagate `Symbol.return_range` into `IR_CALL` destinations,
  so e.g. `bsr_result()` returning [0,31] still appears as unknown to
  the IR-side VRP.

Both pre-existing per the file's header comment.

### `parser.c:2415` returns wrong NODE kind on funcptr-name error path

`parse_func_or_var` returns `new_node(p, NODE_VAR_DECL)` after an
"expected name in function pointer declaration" error, but the sibling
success branch returns NODE_GLOBAL_VAR. Already on an error path
(`p->had_error = true`), so downstream passes bail; not actively
miscompiling, but the inconsistency makes for subtle invariant traps
if someone changes the error policy later.

### Audit-matrix false positives

`tools/audit_matrix.sh checker.c` reports 16 missing context-check
gaps. Spot-checking lines 7186, 7231, 9378-9423 shows the actual
context checks live in different functions (`check_stmt`) than the
walker (`scan_func_props`) the script grepped. The tool needs a
filter that ignores walker functions, or its 16 "BUG:" lines should
be treated as known false positives.
