# Audit 2026-06-06 — ZER Compiler Silent-Gap Sweep

## Scope

Codebase audit focused on the IR-only safety surface (no `zercheck.c`,
which is now a 150-line shim). Files heavily read:

- `zercheck_ir.c` (4145 lines) — production CFG-based safety analyzer
- `ir_lower.c` (3252 lines) — AST→IR lowering, where pre-fix mutations live
- `emitter.c` (10959 lines) — emit-C from IR; safety wrapper emission
- `checker.c` (14125 lines) — type checks, intrinsic dispatch, escape, etc.
- `parser.c` (2987 lines) — token-driven intrinsic-arg classification
- `src/safety/*` (33 files) — extracted predicates verified by VST

Parallel exploration agents were dispatched across the four largest
files for breadth; every finding was independently verified by writing
a minimal `.zer` reproducer and re-running `zerc` end-to-end. Findings
that did not reproduce (despite the agent's claim) were discarded.

## Confirmed silent gaps — closed in this session

Six gaps were confirmed by reproducer. The four most tractable were
fixed; tests are now in `tests/zer_fail/`. The reproducer files for
the two remaining (open) gaps live in `tests/audit_2026_06_06/`.

### GAP-A — Cross-function FuncSummary silently skipped compound args

**Severity:** CRITICAL (silent cross-function double-free)

**Root cause:** `zercheck_ir.c` IR_CALL summary-application loop only
resolved `arg_local` for the decomposed-temp path and a NODE_IDENT
passthrough path. The IR pre-decomposes `free_handle(c.h)` into a
FIELD_READ temp + CALL with the temp as `call_arg_locals[0]`; the
temp has no tracked handle, so the summary fired against an unfilled
slot and the source compound `c.h` retained its ALIVE state. A
subsequent `slab.free(c.h)` then saw a fresh handle and compiled
cleanly.

**Witness shape:** the extern_free path at lines 2902-2942 already
called `ir_extract_compound_key`, so direct `free(c.h)` was caught;
the summary path was the only one that missed compound resolution.

**Fix:** mirror the extern_free pattern. Even when the decomposed
temp resolved, re-inspect `inst->args[pi]` via
`ir_extract_compound_key`. Prefer the compound-resolved handle when
it actually finds tracked state; otherwise fall back to the
decomposed temp for auto-registration. `zercheck_ir.c:2961-3046`.

**Regression test:** `tests/zer_fail/audit_summary_compound_double_free.zer`.

### GAP-C — IR spawn handler silently skipped compound move-struct args

**Severity:** HIGH (silent use-after-move on threading boundary)

**Root cause:** `zercheck_ir.c` IR_NOP+NODE_SPAWN handler (line 1599-
1622, dispatched here because `ir_lower.c` lowers NODE_SPAWN to IR_NOP
with the AST node passed through) explicitly checked
`arg->kind != NODE_IDENT` and skipped compound spawn arguments. A
`spawn worker(b.t)` where `b.t` was a move struct never transitioned
to IR_HS_TRANSFERRED.

**Witness shape:** `spawn worker(t)` with a bare local `t` WAS caught,
proving the gap was localized to the NODE_IDENT-only filter.

**Fix:** route the spawn-arg sweep through `ir_extract_compound_key`,
auto-register the compound when the underlying field type is itself a
move struct, then mark TRANSFERRED. Compound-path field walk uses
`SField` traversal. `zercheck_ir.c:1592-1683`.

**Regression test:** `tests/zer_fail/audit_spawn_compound_move_struct.zer`.

### GAP-D — Destructor-name heuristic regression in IR path

**Severity:** HIGH (silent UAF on non-void bodyless destructors)

**Root cause:** the pre-migration AST analyzer (`zercheck.c`,
now-deleted) called `name_looks_like_destructor` to widen the bodyless
heuristic so non-void return types like `i32 destroy_resource(*R)`
were classified as freeing their first arg. `docs/refactor_ir.md:2293`
claimed this had been "bundled into `ir_is_extern_free_call`" but the
IR helper only matched literal "free" + void-return + bodyless. Result:
calls to user-named destructors with non-void return type were not
detected as freeing, producing a misleading "handle never freed" leak
warning AND silently allowing the post-free use to compile.

**Fix:** restored `ir_name_looks_like_destructor` in `zercheck_ir.c`
with the original 12-keyword substring list (free, destroy, close,
release, delete, dispose, drop, cleanup, deinit, fini, shutdown,
term). Widened `ir_is_extern_free_call` to accept non-void bodyless
functions when the name matches. `zercheck_ir.c:793-862`.

**Regression test:** `tests/zer_fail/audit_destructor_nonvoid_uaf.zer`.

### GAP-E — Parser dropped typedef name as intrinsic type argument

**Severity:** MEDIUM (confusing diagnostic, not a safety hole)

**Root cause:** `parser.c:911-917` only included `@cast`, `@bitcast`,
`@truncate`, `@saturate` in `force_type_arg`. For `@ptrcast(TypedefName,
ap)` etc., the typedef name lexed as TOK_IDENT and fell through the
`(p->current.type != TOK_IDENT || force_type_arg)` discriminator,
becoming an expression argument. The checker handler then saw
`node->intrinsic.type_arg == NULL`, fell through to `result = ty_void;`,
and the var-decl init failed at the use site with the confusing
"cannot initialize 'p' of type '*T' with 'void'" message.

**Fix:** extended `force_type_arg` to include `@ptrcast`, `@pun`,
`@inttoptr`, `@container`. `parser.c:911-925`.

The reproducer of the original surface symptom is no longer interesting
(it now compiles per the design); the distinct-typedef qualifier-strip
case below (GAP-F) is the residual semantic gap that survived the
parser fix.

## Confirmed silent gaps — still open

### GAP-B — Extern alloc fused with var-decl + orelse drops tracking

**Severity:** CRITICAL (silent UAF in C-interop boundary, the very
case `*opaque` round-trip is supposed to catch)

**Root cause:** IR_ASSIGN handler in `zercheck_ir.c:2028-2263` unwraps
the orelse and dispatches on builtin methods (IRMC_*) or
`dest type == TYPE_HANDLE`. Extern alloc returning `?*T` for a non-
Handle pointer dest falls between both branches. The IR_CALL-level
extern recognition at lines 2828-2900 doesn't fire because the outer
instruction is IR_ASSIGN, not IR_CALL.

**Witness:** the two-step form
```
?*Res maybe = my_alloc();
*Res  p     = maybe orelse return;
```
DOES catch the UAF, proving the gap is in the fused var-decl + orelse
+ extern alloc shape.

**Reproducer:** `tests/audit_2026_06_06/audit_extern_alloc_orelse_uaf.zer`.

**Suggested fix:** Mirror the `IRMC_NONE + TYPE_HANDLE` branch
(line 2233-2263) for non-Handle pointer types when the unwrapped RHS
matches `ir_is_extern_alloc_call`. Care: do not double-register
when a downstream IR_CALL also sees the alloc.

### GAP-F — `@ptrcast` qualifier-strip ignores distinct typedef target

**Severity:** MEDIUM (silent const/volatile launder through distinct
typedef)

**Root cause:** `checker.c:6095-6101` checks
`result->kind == TYPE_POINTER && !result->pointer.is_const` without
unwrapping `result`. For a `distinct typedef *u32 PlainPtr;` target,
`result->kind` is TYPE_DISTINCT, so the strip-check silently passes.
The downstream `can_implicit_coerce` sees `*u32 = *u32` (after
unwrap) and accepts the assignment. The const qualifier on the source
was effectively laundered through the distinct typedef.

**Reproducer:** `tests/audit_2026_06_06/audit_ptrcast_distinct_const_strip.zer`.

**Suggested fix:** `Type *tgt_eff = type_unwrap_distinct(result);` at
the start of the const-strip and volatile-strip blocks (and the
`check_volatile_strip` helper). Same pattern as the source-side
`eff = type_unwrap_distinct(val_type)` at line 6088. Verify the same
unwrap is also applied in the @pun handler at line 6154+.

## Findings that turned out to be false positives

Each agent surfaced a handful of plausible-looking gaps that did
not reproduce. Documenting them keeps future audits from re-treading
them.

- **"NODE_VAR_DECL missing auto-zero for `?u32`"** — ZER's "every
  variable initialized to zero" rule is implemented at the emitter
  (`_zer_opt_u32 maybe_val = {0};`). Verified by hand: `?u32 v; if
  (v) |x| { ... }` declares the optional zero-initialized via the
  emitter's `= {0}` pattern, never reads garbage. Lowering does not
  need to emit an explicit IR_LITERAL.

- **"Async yield doesn't populate state struct"** — `_zer_async_*_poll`
  is emitted exactly per `self->local = self->_zer_t0;` before
  `self->_zer_state = 1; return 0;`, then resume reads
  `self->local`. Verified by inspection of generated C from a yield
  test.

- **"`orelse return` skips defers"** — checked the lowered IR and the
  emitted C for `defer puts("X"); x = maybe() orelse return; ...`.
  The defer block IS emitted before the return on the orelse-fallback
  path (`_zer_bb2:; puts("defer fired"); return 0;`). One cosmetic
  oddity: the fall-through to `_zer_bb3:` after the defer-fired
  return is unreachable code (gcc -O2 elides it). Not a safety bug.

- **"TYPE_DISTINCT bypass in NODE_ASSIGN const/volatile checks"** —
  reproducer shows `can_implicit_coerce` does catch the strip even
  when the local raw-`kind == TYPE_POINTER` check misses. Per
  CLAUDE.md "Confirmed NOT Bugs", these are belt-and-suspenders
  diagnostics; the worst-case is a less precise error message. The
  distinct-typedef variant on @ptrcast (GAP-F above) IS a real gap
  because `can_implicit_coerce` does not re-check qualifiers.

- **"@bitcast width validation skipped on distinct"** — Width check
  uses `type_width` which already unwraps. Verified by `@bitcast(u32,
  big)` and `@bitcast(SmallDistinct, big)` both producing the same
  "width mismatch" diagnostic.

## Methodology notes

- **Spawn-then-verify protocol works.** Three parallel agents found a
  combined 35+ candidate gaps; verification reduced these to ~5 real
  ones plus the parser/destructor regressions I found independently.
  The cost of the verification step is small (one reproducer + grep)
  compared to the cost of acting on a hallucinated gap.

- **`ir_extract_compound_key` is the recurring unifier.** Whenever a
  safety walker filters on `arg->kind == NODE_IDENT`, the pattern
  almost certainly hides a compound-shape silent miscompile. Audit
  every remaining NODE_IDENT-only filter in `zercheck_ir.c` against
  the compound-key resolution call sites. Easy mechanical sweep:
  `grep -n 'kind == NODE_IDENT' zercheck_ir.c` (~36 sites today;
  triage which ones could be widened without over-warning).

- **The `name_looks_like_destructor` regression survived the AST→IR
  migration's "Parity" status table.** Cross-check claims in
  `docs/refactor_ir.md` against actual helper-body equivalence
  (not just helper presence) for any future migration.

- **The parser `force_type_arg` list is a known maintenance hazard.**
  Any new type-receiving intrinsic added in the future MUST be added
  to this list or it will silently degrade to `ty_void`. Consider
  inverting the discriminator: instead of an allow-list, parse any
  TOK_IDENT first arg as type if the next significant token is `,`
  and the typedef resolves to a real type. (Higher risk, more lookahead;
  not done in this session.)
